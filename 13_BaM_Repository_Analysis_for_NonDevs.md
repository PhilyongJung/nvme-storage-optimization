# BaM (Big Accelerator Memory) 저장소 비개발자용 분석

> **분석 대상**: github.com/ZaidQureshi/bam
> **한줄 요약**: GPU가 CPU의 도움 없이 SSD에서 직접 데이터를 가져오는 기술

---

## 1. 이 프로젝트가 무엇인가?

### 쉽게 말하면

> 큰 도서관(SSD)에서 책(데이터)을 찾아야 하는데,
> 원래는 반드시 **사서(CPU)**에게 부탁해야 했다.
> BaM은 **독자(GPU) 본인이 직접 서가에 가서 책을 꺼낼 수 있게** 만드는 시스템이다.

| 항목 | 내용 |
|------|------|
| **이름** | BaM — Big Accelerator Memory |
| **만든 사람** | NVIDIA, IBM, 일리노이대, 버팔로대 공동 연구팀 |
| **발표** | ASPLOS 2023 (컴퓨터 구조 분야 최고 학회) |
| **GitHub** | github.com/ZaidQureshi/bam |
| **Stars** | 217 |
| **라이선스** | BSD 2-Clause (자유롭게 사용 가능) |
| **언어** | CUDA / C++ / C |
| **상태** | 연구용 프로토타입 (프로덕션 아님) |

### 왜 만들었나?

| 문제 | 설명 | 비유 |
|------|------|------|
| GPU 메모리가 부족하다 | AI 학습 데이터가 수 TB인데 GPU 메모리는 80~192GB | 작업대(GPU 메모리)가 너무 작아서 재료(데이터)를 다 올려놓을 수 없다 |
| SSD에 데이터가 있지만 GPU가 직접 못 가져온다 | 전통적으로 CPU가 중간에서 SSD→CPU메모리→GPU 순서로 옮겨줘야 한다 | 공장에서 부품이 필요한데, 반드시 사무실(CPU)을 거쳐야 창고(SSD)에서 가져올 수 있다 |
| CPU가 병목이 된다 | GPU는 수만 개 작업자가 동시에 일하는데, CPU는 수십 명뿐. 창구가 막힌다 | 작업자 10,000명이 동시에 부품을 요청하는데, 사무실 직원이 8명뿐 |

**BaM의 해결책**: GPU 작업자가 직접 창고(SSD)에 가서 부품(데이터)을 가져오게 한다. 사무실(CPU)은 처음에 열쇠만 건네주고, 이후에는 관여하지 않는다.

## 2. 어떻게 동작하는가?

### 공장 비유로 이해하는 BaM

```
┌─────────────────────────────────────────────────────────────────┐
│                    🏭 BaM 공장 전체 구조                         │
│                                                                 │
│  ┌─────────────────────────────────────────────────────┐       │
│  │  4층: 작업 현장 (GPU 커널)                            │       │
│  │  "작업자(GPU 스레드)가 data[i]로 부품을 요청한다"       │       │
│  │  → 선반에 있으면 바로 사용 (캐시 히트)                  │       │
│  │  → 없으면 자동으로 창고에서 가져온다 (캐시 미스)        │       │
│  └──────────────────────┬──────────────────────────────┘       │
│                         │                                       │
│  ┌──────────────────────▼──────────────────────────────┐       │
│  │  3층: 부품 선반 (GPU 소프트웨어 캐시)                  │       │
│  │  "자주 쓰는 부품을 작업대 옆 선반에 보관"               │       │
│  │  → 선반이 꽉 차면 오래된 부품을 치우고 새 부품 배치     │       │
│  │  → 수정된 부품은 창고에 다시 보냄 (write-back)         │       │
│  └──────────────────────┬──────────────────────────────┘       │
│                         │ 선반에 없을 때                         │
│  ┌──────────────────────▼──────────────────────────────┐       │
│  │  2층: 주문 처리 시스템 (NVMe 큐)                      │       │
│  │  "작업자가 직접 주문서를 작성하고 창고에 보낸다"         │       │
│  │  → 주문서(NVMe 명령)를 접수대(SQ)에 놓음               │       │
│  │  → 벨(doorbell)을 눌러 창고에 알림                     │       │
│  │  → 배달 완료 확인(CQ)을 직접 확인                      │       │
│  └──────────────────────┬──────────────────────────────┘       │
│                         │                                       │
│  ┌──────────────────────▼──────────────────────────────┐       │
│  │  1층: 창고 (NVMe SSD)                                │       │
│  │  "주문서를 받으면 지게차(DMA 엔진)가 부품을 3층으로"    │       │
│  │  → PCIe 고속도로를 통해 직접 GPU 선반에 배달            │       │
│  │  → CPU 사무실을 거치지 않는 직통 경로                   │       │
│  └─────────────────────────────────────────────────────┘       │
│                                                                 │
│  🏢 사무실(CPU)의 역할: 공장 개장 시 열쇠 전달과 초기 설정만     │
│     → 이후 공장 운영(런타임)에는 관여하지 않음 (CPU 0%)         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 단계별 동작 요약

| 단계 | 무슨 일이 일어나는가 | 누가 하는가 | 비유 |
|------|---------------------|-----------|------|
| **0. 초기 설정** | SSD 장치 등록, GPU 메모리 매핑, NVMe 큐 생성 | CPU (1회만) | 공장 개장: 열쇠 전달, 선반 설치, 주문 시스템 세팅 |
| **1. 데이터 요청** | GPU 스레드가 `data[i]`로 데이터 접근 | GPU 스레드 | 작업자가 "부품 #3742번 필요"라고 선반에 손을 뻗음 |
| **2. 캐시 확인** | 선반(캐시)에 해당 데이터가 있는지 확인 | GPU 스레드 | 선반을 둘러보며 부품이 있는지 확인 |
| **3a. 히트** | 데이터가 있으면 바로 반환 | GPU 스레드 | 선반에서 바로 부품을 집어서 작업 재개 |
| **3b. 미스** | 데이터가 없으면 NVMe 읽기 명령 구성 | GPU 스레드 | 주문서를 작성해서 접수대에 놓음 |
| **4. 도어벨** | SSD에 "새 주문이 있다"고 알림 | GPU 스레드 | 접수대의 벨을 눌러 창고에 알림 |
| **5. DMA 전송** | SSD가 데이터를 GPU 메모리로 직접 전송 | NVMe DMA 엔진 | 지게차가 고속도로(PCIe)로 부품을 직접 선반에 배달 |
| **6. 완료 확인** | GPU 스레드가 완료 큐를 확인 | GPU 스레드 | 작업자가 "배달 왔나?" 확인 |
| **7. 데이터 사용** | 데이터를 받아서 연산 수행 | GPU 스레드 | 부품을 받아서 작업 계속 |

### 전통 방식과 BaM 비교

| 비교 항목 | 전통 방식 | BaM |
|-----------|----------|-----|
| **데이터 요청자** | CPU가 대신 요청 | GPU가 직접 요청 |
| **데이터 경로** | SSD → CPU메모리 → GPU메모리 (2번 이동) | SSD → GPU메모리 (1번 직행) |
| **CPU 역할** | 모든 과정에 관여 | 처음 설정만, 이후 0% |
| **비유** | 사서에게 부탁해서 책을 받는다 | 직접 서가에 가서 책을 꺼낸다 |
| **프로그래머 코드** | 파일 열고→읽기→복사→GPU전달 (복잡) | `data[i]` 한 줄 (간단) |

## 3. 왜 이런 구조로 설계되었는가?

### 설계 결정과 그 이유

| 설계 결정 | 왜 이렇게 했는가 | 비유 |
|-----------|-----------------|------|
| **GPU에 소프트웨어 캐시를 둔다** | SSD는 GPU 메모리보다 100~1000배 느리다. 자주 쓰는 데이터를 GPU 메모리에 보관하면 SSD 접근을 줄일 수 있다 | 자주 쓰는 도구를 작업대 서랍에 넣어두면 창고에 갈 필요가 줄어든다 |
| **NVMe 큐를 GPU 메모리에 놓는다** | GPU 스레드가 CPU 도움 없이 직접 SSD에 명령을 보내려면, 주문서 접수대(큐)가 GPU 안에 있어야 한다 | 주문 접수대를 사무실(CPU)이 아닌 작업 현장(GPU)에 설치 |
| **표준 NVMe 드라이버를 쓰지 않는다** | 리눅스 표준 NVMe 드라이버는 CPU가 제어하도록 설계됨. GPU가 직접 제어하려면 커스텀 드라이버가 필요 | 공장 전용 출입구를 만든다. 기존 정문(표준 드라이버)으로는 GPU가 직접 못 들어감 |
| **Warp 단위로 요청을 합친다** | GPU에서 32개 스레드(1 warp)가 같은 데이터를 요청하면, 대표 1명만 SSD에 요청하고 결과를 공유 | 32명이 같은 부품을 원하면, 1명만 창고에 다녀와서 나머지에게 나눠줌 |
| **배열 문법(`data[i]`)으로 접근** | 프로그래머가 SSD I/O를 직접 관리하면 코드가 매우 복잡해진다. 배열처럼 쓰면 SSD 존재를 몰라도 된다 | 수도꼭지를 틀면 물이 나온다. 정수장과 배관 구조를 알 필요 없다 |

### 이런 구조의 장단점

| 구분 | 내용 |
|------|------|
| **장점 1** | CPU 사용률 0% — GPU가 스토리지를 독립 관리하므로 CPU는 다른 일 가능 |
| **장점 2** | 세밀한 접근 — 512바이트 단위로도 효율적 (GDS는 32KB 이상이어야 효율적) |
| **장점 3** | 코드 단순 — `data[i]`로 TB급 데이터를 배열처럼 접근 |
| **장점 4** | 비용 절감 — 비싼 GPU 메모리 대신 저렴한 SSD로 대규모 데이터 처리 (최대 21.7배) |
| **단점 1** | 표준 파일시스템 사용 불가 — SSD를 통째로 점유, 다른 프로그램과 공유 안 됨 |
| **단점 2** | 특수 하드웨어 필요 — 데이터센터 GPU(V100/A100/H100)만 지원, 일반 GeForce 불가 |
| **단점 3** | 보안 기능 비활성화 — IOMMU를 꺼야 해서 멀티 사용자 환경에 부적합 |
| **단점 4** | 연구 수준 — NVIDIA 공식 지원 없음, 최신 리눅스 커널과 호환 문제 가능 |

## 5. 설계 철학과 장단점

### 설계 철학 3가지

| 철학 | 설명 | 비유 |
|------|------|------|
| **"CPU를 빼자"** | CPU가 병목이므로 런타임에서 완전히 제거. GPU가 SSD를 자율적으로 제어 | 중간 관리자를 없애고 작업자가 직접 의사결정 |
| **"프로그래머에게 투명하게"** | `data[i]`로 접근하면 SSD인지 GPU 메모리인지 신경 쓸 필요 없음 | 수도꼭지를 틀면 물이 나온다 — 정수장 구조를 몰라도 됨 |
| **"GPU의 병렬성을 I/O에도 활용"** | 수만 GPU 스레드가 동시에 I/O를 발행하여 SSD 대역폭을 최대한 활용 | 작업자 10,000명이 동시에 주문 → 창고 지게차가 쉴 틈 없이 가동 |

### 종합 장단점

| 구분 | 항목 | 상세 |
|------|------|------|
| **장점** | 성능 | SSD 10개로 45.8M IOPS, 22.9 GB/s (PCIe 90% 활용) |
| **장점** | 비용 | 1TB GPU메모리 서버 대비 21.7배 저렴한 SSD 구성으로 동등 이상 성능 |
| **장점** | CPU 해방 | 런타임 CPU 0% → CPU를 다른 용도로 활용 가능 |
| **장점** | 세밀한 접근 | 512바이트 단위도 효율적 (GDS는 32KB+ 필요) |
| **장점** | 코드 단순성 | `data[i]` 한 줄로 TB급 데이터 접근 |
| **단점** | 호환성 | 데이터센터 GPU(V100/A100/H100)만, GeForce 불가 |
| **단점** | 설치 복잡 | 커스텀 커널 모듈, IOMMU OFF, BIOS 설정 변경 필요 |
| **단점** | 파일시스템 없음 | SSD를 통째로 점유, 일반 파일로 접근 불가 |
| **단점** | 안정성 | 연구 프로토타입 — 최신 커널(6.x) 호환 문제 보고됨 |
| **단점** | 보안 | IOMMU 비활성화로 장치 간 메모리 격리 없음 |

### BaM vs 다른 기술 한눈에 비교

| 비교 항목 | 전통적 방식 | NVIDIA GDS (공식) | BaM (연구) |
|-----------|-----------|------------------|-----------|
| **누가 SSD에 요청?** | CPU | CPU | **GPU** |
| **CPU 관여** | 전 과정 | 제어만 | 초기화만 |
| **데이터 이동 경로** | SSD→CPU→GPU (2번) | SSD→GPU (1번) | SSD→GPU (1번) |
| **프로그래머 코드** | 파일 읽기 + memcpy | cuFileRead() | **data[i]** |
| **최소 효율 크기** | 수백 KB | 32KB | **512B** |
| **실제 사용 가능?** | 가능 | 가능 (프로덕션) | 연구 단계 |
| **비유** | 사서에게 부탁 | 사서가 직배달 | **직접 서가로** |
